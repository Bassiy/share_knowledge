# 仕様書の使い捨て戦略

## 概要
仕様書はコードと常に同期させる「Single Source of Truth」にはなれない。
新規開発の初期にだけ書き、実装完了後は捨てる使い捨て設計が AI 時代の正解。

## 理解したこと
- 「仕様駆動開発（SDD）」は仕様書とコードを常に同期させる開発スタイルだが、これは無理ゲー
  - 時間が経つほどコードと仕様書が乖離していく
  - AI はコードと仕様書を「等しく信頼」するので、古い仕様書が混入すると誤ったコードを生成する
- 仕様書の価値は「完成物」ではなく「思考プロセスの補助」にある
  - 新規開発初期：チームの認識統一・AI への指示のために書く（マークダウン1〜2ページで十分）
  - 実装完了後：捨てる。コードがすべての真実になる
  - 既存コード改修：書かない。コードとテスト自体が仕様

### 場面別の判断基準

| 場面 | 仕様書の扱い |
|------|------------|
| 新規開発初期 | 書く（チーム認識統一・AI指示用） |
| 実装完了後 | 捨てる（PRのdescriptionに残す程度） |
| 既存コード改修 | 書かない（コードを直接読む） |

### 残すべきドキュメントとの区別
- **仕様書（実装後）**：捨てる。コードの劣化コピー
- **ADR（意思決定の記録）**：残す。「なぜ」はコードを読んでもわからない
- **CLAUDE.md・開発手順**：残す。仕様ではなく「開発メタデータ」
- **ビジネス文脈**：コード内のdocstringとして一体化が理想

### Hook による防衛線（補足）
- 仕様書の乖離自体はHookでは防げない（根本は設計思想で解決）
- ただし「古いドキュメントをAIが読み込む行為」はPreToolUse Hookでブロックできる
- 両方組み合わせると多重防衛になる：「使い捨て設計 ＋ 古いファイルの読み込みブロック」

### RDRA との関係（上流の補完）

| フェーズ | 担当 |
|---------|------|
| 要望 → 要求 → 要件 | RDRA（仕様を書く前） |
| 要件 → 仕様 → 設計 | SDD・仕様書 |

- 仕様書を「使い捨て」で書く前に、そもそも何を書くかを RDRA で固める
- 要求が曖昧なまま仕様を書くと、AI が仮定違いで空転しやすい
- Claude Code に Confluence/Slack を漁らせて要求の仮説を出させる手法と相性がよい（→ rdra.md）
