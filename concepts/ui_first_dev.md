# UIファースト開発

## 概要
UIを先に固めることで、設計・実装の無駄を減らす個人開発のワークフロー。

## 理解したこと

### フロー全体

1. **要件の言語化**
   - 自然言語でアプリの概要・目的をアウトプット

2. **暫定技術選定**
   - 現状の要求から使用技術を洗い出す（確定でなくてよい）
   - Gemini Canvas に投げる雛形デザイン用プロンプトを生成する

3. **雛形デザイン作成（Gemini Canvas）**
   - 1〜3個の雛形を作成
   - Gemini側での改良は破綻しやすいので、最初の納得できる状態で止める

4. **デザイン改良（LLM × インタラクティブプレビュー）**
   - 対話 & 即反映で色・レイアウト・構成を固める

5. **フレームワーク変換 & フロントエンド情報の確定**
   - 選定技術に変換してdev serverで対話 & 即反映
   - 詳細設計のリバースエンジニアリングに必要なUI構造・データ表示・インタラクションを固める
   - フロントエンドの「完成」ではなく「詳細設計の入力」として使う

6. **詳細設計（リバースエンジニアリング）**
   - 固まったUIからAPI・データ構造・画面遷移を逆算
   - 知識ベースを引用しながら設計書を作成
   - **責務分離を意識して設計する**：画面表示・データ取得・ビジネスロジックの境界を明確にする
   - **テスト項目を設計と同時に作成する**：何をRequest Spec / Model Spec / Query Specで確認するかを設計書に含める
   - スタブ（ダミーデータ）で動作確認

7. **実装**
   - フロント・バック共に実装
   - コード生成後に決定論的ツール（テスト・Linter）を実行し、エージェントの自己修正ループを回す
   - PostToolUse Hookを設定しておくと、ファイル編集のたびに自動実行される（手動で指示不要）
   - スタブを本物に差し替えて完成

### design.md 変換パターン

Gemini Canvasで生成したコードをそのままClaude Codeに渡すのではなく、**一度design.mdに変換してから渡す**ことで精度とトークン効率が上がる。

**design.mdに書くべきもの（意図のみ・実装詳細は書かない）：**
- カラーパレット（hex値）
- タイポグラフィ（サイズ・ウェイト）
- レイアウト（最大幅・余白規則）
- コンポーネント構成（何があるか・配置）
- トーン・雰囲気（ミニマル・フラット等）

**コードをそのまま渡す問題点：**
- GeminiのコードのクセをClaude Codeが引き継ぎやすい
- フレームワーク差異の読み替えでトークンを消費する

**変換フロー（Geminiコード → design.md）：**
```
Geminiのコード
  ↓
① Geminiに「このコードからdesign.mdを書いて」と依頼
   （自分が書いたコードの意図を一番正確に把握している）
  ↓
② Geminiのdesign.md + Geminiのコード を両方Claudeに渡す
   「実装詳細を除いて意図だけに整理して」
  ↓
完成版design.md → 以後これだけをClaude Codeに渡す
```

「実装の詳細ではなく意図を渡す」ことで、Claude Codeが最適な実装を自分で判断できる。

### なぜUIから始めるか
- UIが見えると「必要なデータ」が具体化され、過剰設計を防げる
- デザインという成果物が設計書のインプットになる（リバースエンジニアリング）
- スタブで動かすことでフロント・バックを分離して進められる
