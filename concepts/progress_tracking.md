# プロジェクト進捗管理パターン

## 概要
`docs/progress.md` に開発ステージとチェックリストを持ち、
`/progress` スキルで一覧表示・更新する。Claude Code プロジェクトの標準パターン。

## 理解したこと
- 個人開発でも「今どこにいるか」を明示しないと Claude が文脈を誤解しやすい
- progress.md はスキルから読まれる「生きたドキュメント」なので捨てない（spec_doc_strategy の例外）
- ステージは「セットアップ → 環境構築 → 機能実装 → 統合確認」の順が汎用的

### ファイル構成

```
docs/progress.md           ← ステージ一覧 + チェックリスト（永続）
.claude/skills/progress.md ← /progress スキル
```

### progress.md の構造

```markdown
## ステージ一覧
| # | ステージ | 状態 | 備考 |
|---|---------|------|------|
| 0 | セットアップ | ✅ 完了 | |
| 1 | 環境構築    | 🔲 未着手 | |

## 各ステージの詳細
### ✅ Stage 0: ...
- [x] 完了タスク

### 🔲 Stage 1: ...
- [ ] 未完了タスク

## 次のステップ
最初の 🔲 ステージの具体的な手順
```

### /progress スキルの動作
1. `docs/progress.md` を読む
2. ステージ一覧テーブルを表示
3. 現在ステージ（最初の🔲）の詳細チェックリストを表示
4. 「更新しますか？」と聞いて、返答があれば即編集

### ファイル分割の判断基準
- **原則：1ファイルで管理する**
  - `/progress` スキルが1回の Read で完結する
  - Claude が全体像を把握するコストが低い
- **分割するタイミング：1ステージが膨らんできたとき**（デバッグメモ・調査ログが50行超など）
  ```
  膨らむ前: docs/progress.md（1ファイルで完結）
  膨らんだら:
    docs/progress.md        ← 概要テーブル + 次のステップのみ残す
    docs/progress/stage-3.md ← 詳細ログを切り出す
  ```
- 小規模な個人開発（6〜8ステージ）では分割不要なことがほとんど

### spec_doc_strategy との違い
- 仕様書（spec.md）→ 実装後に捨てる
- progress.md → プロジェクト完了まで更新し続ける。ADR に近い扱い
