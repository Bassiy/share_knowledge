# vibe_coding_security

## 概要
生成AIでコードを書く「バイブコーディング」スタイルにおいて、本番リリース前に守るべきセキュリティ・品質原則。

## 理解したこと

### なぜ必要か
- AIは「動くコード」を生成するが、「安全なコード」「壊れないコード」は保証しない
- バイブコーディングは開発速度を上げる一方、基礎知識なしに本番リリースされるリスクを上げる

### 6つの原則

**1. セキュリティ（最重要）**
- APIキーはコードに書かない → 環境変数で管理
- ログイン（認証）≠ データアクセス許可（認可）。RLSなどで明示的に制御する
- 入力検証はサーバーサイドで必ず行う（クライアントは信用しない）
- ファイルアップロードは拡張子・MIMEタイプ・サイズを検証する
- IaaSは避け、SaaS型ビルダー（Lovable・Bolt.newなど）の利用を検討する

**2. コスト管理**
- 従量課金モデルは事前見積もりと上限設定が必須
- リリース後も定期的な請求確認を続ける

**3. 法的確認**
- 利用規約・個人情報保護法・ライセンス互換性を事前チェック

**4. データ設計**
- スキーマ設計はリリース前に固める（後から修正コストが最大になる）

**5. テスト戦略**
- 「動く」と「壊れない」は別。最低限のテストを書く

**6. 先人の知見を活かす姿勢**
- 50年分の失敗パターンを学ぶ。ソフトウェア業界の知見は蓄積されている

### 認証と認可の違い（重要な落とし穴）
```
認証（Authentication）: 「あなたは誰ですか？」= ログイン
認可（Authorization）:  「あなたはこれにアクセスできますか？」= データ制御

ログインできる ≠ 全データにアクセスできる　← ここを混同するとデータ漏洩
```
