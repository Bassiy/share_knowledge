# SSH（Secure Shell）

## 概要
サーバやネットワーク機器にリモート接続してCUI操作するためのプロトコル・プログラム。ポート22番。

## 理解したこと
- **何をするもの**：ネットワーク越しにサーバや機器を遠隔でコマンド操作する
- **前身はTelnet**：Telnetは暗号化なし・認証が弱かった。SSHはその改善版
- **2つのセキュリティ機構**：
  - **通信の暗号化**：通信内容が盗聴されない
  - **公開鍵認証**：パスワード総当たりが通用しない
- **公開鍵認証の流れ**：
  1. クライアントの公開鍵をサーバに事前登録（漏れても問題ない）
  2. 接続時にクライアントが秘密鍵で署名 → サーバが公開鍵で検証 → 認証成立
- **鍵の使い方の2方向**：
  - 認証：秘密鍵でロック（署名）→ 公開鍵で検証（クライアントが秘密鍵を持つ）
  - 暗号化：公開鍵でロック → 秘密鍵で復号（サーバが秘密鍵を持つ）

## HTTPSとの比較（Git接続での使い分け）

| | HTTPS | SSH |
|---|---|---|
| 暗号化の仕組み | HTTP + TLS（ssl_tls.md） | SSH独自の暗号化 |
| 実装ライブラリ | OpenSSL or GnuTLS | OpenSSH |
| 認証 | PAT（パスワード的なトークン） | 公開鍵認証 |
| ポート | 443 | 22 |

- HTTPSはTLSを使う。TLSの実装にはOpenSSLとGnuTLSがあり、Ubuntuデフォルト GitはGnuTLSを使用
- GnuTLSはバッファ溢れ時に再試行せず接続を切断する既知の弱点がある
- Gitは大量ファイルを「packfile」という1塊にまとめて送るため、GnuTLSと相性が悪い
- SSHはOpenSSHを使い、GnuTLSを一切通らないためこの問題を根本回避できる
- さらにSSHはストリーム接続（つなぎっぱなしで流す）のでチャンク分割のオーバーヘッドが少ない
- 認証面でも、HTTPS（PAT=パスワード的なトークン）より SSH鍵認証の方が秘密鍵がPC外に出ない分安全
