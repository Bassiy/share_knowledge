# Harness Engineering

## 概要
AIエージェントが正しいコードを確実に生産できる「環境」を設計するエンジニアリング手法。

## 理解したこと
- エンジニアの仕事が「正しいコードを書く」から「エージェントが正しく書ける環境を設計する」に移行している
- モデルを変えると性能差1ポイント、ハーネスを変えると22ポイント変わる。環境設計の方が効く
- 核心は「決定論的ツール（Linterなど）でエージェントの自己修正ループを回すこと」
- ドキュメントは腐る。テストとADRを仕様の正とする
- CLAUDE.mdは50行以内・ポインタ集にする（150超えると後半が無視される）
- フィードバック速度の原則: PostToolUse（ms）> pre-commit（s）> CI（min）> human review（h）。速いほど修正コストが下がる
- Linterのエラーメッセージに修正手順を埋め込むと、エージェントが自分で直せる
- Claude Code（Workshop型）とCodex（Closeted型）は設計思想が異なり、2026年の定石はClaudeで計画・Codexで並列実行
- 【実践例】Claude Code単独でSaaSを構築する場合のハーネス設計
  - コンテキスト投入：仕様書PDF＋DBスキーマをAgentに読み込ませる
  - Agentモードで自律ループ（生成→テスト→エラー検知→自己修正）を回す
  - 人間のレビューはRLS（セキュリティ）のみ
  - デプロイはスマートフォンからGitマージ承認→Vercel自動デプロイ
  - 障害対応：監視ボット→Claude Codeエージェントへ通知→自動修正
  → 「人間は何をレビューするか」を絞り込むことがハーネス設計の本質

### RL環境との構造的同一性（2026年の気づき）
- ハーネスエンジニアリング（本番）とRL環境（学習）は「環境設計が成果を決める」という思想が同じ
- RL環境の構成：`Dataset + Harness + Rubric = Environment`（verifiersライブラリの定義）
- つまり良いハーネスを持つ企業 = 良いRL訓練環境を持つ企業でもある
- **将来予測**：本番ハーネスをそのままRL訓練に転用し、自社プロダクト特化モデルを継続育成する企業が登場する。「環境」自体が売買・ライセンスされる資産になりうる

### ボトルネック：検証可能な報酬（RLVR）
- RL環境への転用が成立する前提は「何が正解か」を自動判定できること
- 数学・コードは正誤が明確 → 環境が作りやすい
- 「良い顧客対応か」「良い設計提案か」などソフトなタスクはRubricが作れずRLに回せない
- ハーネスの売買が先に起こるのは「判定しやすいタスク」から。評価軸の標準化（ORS等）が鍵

### Anthropicの設計哲学
- ハーネスのコンポーネントは「モデルが単体ではできないことへの仮説」
- モデルが進化するほどハーネスは薄くなるべき存在（一時的な補助輪）
- ハーネスが厚い = まだモデルが弱い証拠、とも読める

### ドキュメントハーネス：3層アーキテクチャ（2026年の実践例）

「品質はドキュメントの精度ではなく、検証基盤で決まる」

AIへの指示書（DESIGN.md / CLAUDE.md）はプロジェクトが育つほど腐る。
これを防ぐために仕様を3層に分離し、自動検証を組み込む。

```
Layer 1（哲学・原則）: DESIGN.md
  └── ブランド哲学・設計原則・クイックリファレンス
      ツール非依存。Claude以外でも使い回せる

Layer 2（機械可読な仕様）: contracts/
  ├── components/*.json（コンポーネント仕様）
  └── prohibitions/*.json（禁止ルール）
      Single Source of Truth

Layer 3（検証基盤）: harness
  ├── 静的チェック（命名・クラス使用）
  ├── ドリフト検出（仕様 vs コードの乖離）
  ├── リアルタイムフック（保存時に即検証）
  └── CIパイプライン（PR時に自動実行）
```

- Layer 1・2 だけでは「よく整理されたドキュメント」で終わる
- Layer 3（ハーネス）があって初めて「壊れたら気づく」仕組みになる
- 人間が追いきれなくなったとき（50コンポーネント規模〜）に導入を検討するのが適切
