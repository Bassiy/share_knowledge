# SSL/TLS

## 概要
ネットワーク通信に暗号化・改ざん検出・認証を提供するセキュリティプロトコル。HTTPSの基盤となる。

## 理解したこと

### 3つの機能
| 機能 | 内容 |
|------|------|
| 暗号化 | 通信を盗聴されないようにする |
| 改ざん検出 | 通信途中でデータが書き換えられた場合に検出する |
| 認証 | 通信相手が本物であることを確認する |

### レイヤー上の位置
- TCP/IPモデル：アプリケーション層
- OSIモデル：第6層（プレゼンテーション層）あたりが妥当
  - 第5層セッション層：通信の開始〜終了を管理
  - 第6層プレゼンテーション層：データ形式を整える → TLSの暗号化と近い役割

### ネゴシエーション
- 使用する暗号化アルゴリズムは**セッション確立のたびに**決定される
- クライアントが候補リストを提示 → サーバーが安全性の高いものを選択
- TLS 1.3などの最新技術では、過去に通信した相手に対してネゴシエーションをスキップし即座にデータ送信できる（0-RTT）

### 公開鍵暗号（補足）
「公開鍵」と「秘密鍵」のペアを使う暗号方式。公開鍵で暗号化したデータは対応する秘密鍵でのみ復号できる。
TLSでは主に**共通鍵の安全な受け渡し**に公開鍵暗号が使われる。

### デジタル署名とPKI（補足）
秘密鍵で署名を生成し、公開鍵で検証する仕組み。
- 秘密鍵を持つ本人しか署名を生成できない → **認証**
- 署名が一致しなければ改ざんとわかる → **改ざん検出**

TLSでは「デジタル証明書」にCA（認証局）がデジタル署名することで正当性を保証する。
CAはルートCA → 中間CA → サーバー証明書という階層構造（PKI）を形成しており、ブラウザはこの信頼チェーンを辿って証明書を検証する。

### HTTPSにおける2段階の認証
1. **サーバー認証**（PKI）：CAの証明書でサーバーが本物かを確認
2. **ユーザー認証**：パスワード・OAuth等でユーザーが本物かを確認

①で安全な通信路を確立してから②が行われる。①がないと偽サイトにパスワードを送るリスクがある。
