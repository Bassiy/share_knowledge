# OAuth 2.0

## 概要
パスワードを直接渡さず、認証サーバーが発行した「期間限定のアクセストークン」を使って安全にリソースへアクセスする認可フレームワーク。

## 理解したこと

### 基本の仕組み

```
従来：アプリ ←パスワード直接→ サーバ

OAuth2.0：
アプリ → 認証サーバ（パスワードでトークン発行）
アプリ → サーバ（トークンを送る）
サーバ → 認証サーバ（トークンを問い合わせ・検証）
```

### アクセストークン と リフレッシュトークン

| | アクセストークン | リフレッシュトークン |
|---|---|---|
| 役割 | リソースへのアクセス許可証 | 期限切れ後に新しいアクセストークンを取得 |
| 有効期限 | 短命（30分〜1時間程度） | 比較的長い |
| 使い回し | 期間内は何度でも使える | 使うたびに新しいものに交換する設計が多い |

- 「1回きりの使い捨て」ではなく「賞味期限付きチケット」が正確なイメージ

### スコープによる権限の限定
- トークンには「メール読み取りのみ」など操作範囲（スコープ）が設定される
- パスワードは「全権」だが、トークンは「特定のドアの鍵」
- 盗まれてもアカウントのパスワード変更などはできない

### 認証（Authentication）と認可（Authorization）の分離
- **認証**：「あなたは本人ですか？」を確認すること
- **認可**：「この操作を許可します」と権利を与えること
- OAuth2.0 は認可のための仕組み
- アカウント設定変更などには別途「認証（再ログイン・MFA）」が必要 → トークンだけでは乗っ取れない

### 認証サーバー内の2つの窓口
Googleなどの認証サーバーには論理的に2つの窓口がある。

| 窓口 | 担当 | やること | 結果 |
|---|---|---|---|
| 認証窓口 | Googleの独自システム | ID/パスワード・2FA・指紋認証などで人間を確認 | 「この人はBassiyさん」という確信 |
| 認可窓口 | OAuth 2.0 | 本人確認済みを前提に「このアプリにメール読み取り権限を渡していい？」と確認 | アクセストークンを発行 |

- 認証が先に完了していないと、認可窓口は動かない

### なぜ「認可だけをOAuthで共通化」するのか
- **認証は各社の自由**：GoogleはパスワードやMFA、AppleはFaceIDなど、本人確認の手段は各社が独自に進化させればよい
- **認可は共通ルールにしたい**：トークンの渡し方・検証の手順がバラバラだと、開発者はGoogle用・Microsoft用・Slack用と別々の実装を書く必要が生じる
- OAuth 2.0という共通規格があることで、一度覚えれば異なるサービスとも同じ手順で連携できる

### なぜ漏洩してもダメージが少ないか
1. スコープで操作が限定されている
2. 短命なので長期的な悪用ができない
3. 認証サーバー自体を騙して本人になりすますことはできない

### 最小権限の原則（Principle of Least Privilege）
必要最小限の権限だけを渡すことで、事故時の被害を最小化する思想。OAuth2.0のスコープ設計はこの思想を体現している。APIを作る側になったとき「どの操作にどのスコープを割り当てるか」の判断基準になる。
