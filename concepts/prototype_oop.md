# プロトタイプベースOOP

## 概要
JavaScriptが採用するOOPの仕組み。クラスを定義してインスタンスを生成するのではなく、オブジェクトが別のオブジェクトを参照して委譲するチェーン構造を持つ。

## 理解したこと

### クラスベースとの根本的な違い

| | クラスベース（C#） | プロトタイプベース（JS） |
|---|---|---|
| メソッドの解決 | コンパイル時に確定（vtable） | 実行時にチェーンを辿る |
| 構造の変更 | 実行中はできない | 実行中でもできる |
| 型の確定 | 静的 | 動的 |

### プロトタイプチェーン

`hero.attack()` を呼ぶとき：

```
1. hero 自身に attack があるか？ → ない
2. hero.[[Prototype]]（= Character.prototype）に attack があるか？ → ある → 実行
```

チェーンの終端は `null`。見つからなければ `undefined`。

```
hero ──→ Character.prototype ──→ Object.prototype ──→ null
```

### コピーではなく委譲

- **データ（name, hpなど）**：インスタンスごとに持つ
- **メソッド（attack など）**：prototypeに1つだけ存在し、全インスタンスがアドレスで参照する

参照型変数がメモリ上のアドレスで管理されるのと同じ構造。インスタンスはメソッドのコードを持たず、prototypeのアドレス（8バイト）だけを持つ。

**メモリ効率の比較（1000体・メソッド1つ=1KBの場合）：**

| | メソッドのメモリ使用量 |
|---|---|
| コピー | 1000体 × 1KB = 1,000KB |
| 参照（プロトタイプ） | 1KB + 1000 × 8B ≈ 9KB |

約**111倍**の差。インスタンスが増えるほど差は広がる（10,000体なら約1,111倍）。

### 実行時書き換えの可能性

prototypeはただのオブジェクトなので実行中に書き換えられる。全インスタンスに即座に影響する。

```js
Character.prototype.attack = function() { console.log("新しい攻撃！"); };
// hero1もhero2も全員変わる
```

これが可能なのは「呼ぶたびに実行時にチェーンを辿る（動的束縛）」から。ポリフィルはこれを意図的に使った例。

### DIPとの違い

| | DIP（C#） | prototype書き換え |
|---|---|---|
| 影響範囲 | 1インスタンス | 全インスタンス |
| 抽象への依存 | あり（interface経由） | なし |
| 設計の意図 | 柔軟な差し替え | 基本やらない |
