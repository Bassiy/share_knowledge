# REST API

## 概要
WebのHTTP通信の仕組みを活かしてAPIを設計するための「思想・ガイドライン」。特定の製品ではなく、使いやすいAPIを作るためのお作法。

## 理解したこと

### 慣用的な定義
ネットワーク経由で機能を呼び出し、XMLまたはJSON形式で結果を返すスタイル。

### 厳密な定義（OSI的な理想より）

| # | 原則 | 内容 |
|---|------|------|
| 1 | ステートレス | サーバーが前の状態を覚えない。1回完結の関数呼び出し |
| 2 | 操作手段が定義されている | GET（取得）・POST（保存）など標準の動詞で操作が決まっている |
| 3 | 一意に識別できる | URLという住所があれば世界中からリソースを特定できる |
| 4 | ハイパーメディア | レスポンスに「次に叩けるURL」のリンクが含まれる |

※ 4つすべてを満たさないとREST APIでないわけではない。あくまで理想の設計指針。

### 設計思想である理由
- **強制ではなく推奨** — ルールを破っても通信は動く。ただし従うと使いやすいAPIになる
- **WebのポテンシャルをAPIに応用** — ブラウザとサーバーの既存の仕組み（GETは取得、URLは場所）をそのまま使う
- **他の思想との比較** — GraphQL（1URLでクエリにより欲しい情報を取得）と比べると「設計思想の違い」であることがわかる。優劣ではなく考え方の違い

### APIとの関係
REST APIはAPIという大きなカテゴリの中の「一種（人気なスタイル）」。APIはネットワーク越しだけでなく同じPC内（Win32 API, UnityのAPIなど）も含む広い概念。
