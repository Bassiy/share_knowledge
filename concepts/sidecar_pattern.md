# サイドカーパターン

## 概要
メインのコンテナと一緒に補助用コンテナを同じ実行単位（ECSタスク等）に同居させる設計パターン。

## 理解したこと
- バイクのサイドカー（側車）のイメージ。メインの隣にくっついて補助する
- ECSの「タスク」という単位の中に複数コンテナを入れられる
- APIコンテナ（メイン）+ Workerコンテナ（サイドカー）の構成が典型例
- WorkerはSQSからメッセージを取り出して非同期処理を担当する
- グレースフルシャットダウンと組み合わせることでデプロイ時のデータ整合性を保つ

## 構成イメージ
```
ECSタスク
  ├── APIコンテナ（メイン） → SQSにメッセージ積む
  └── Workerコンテナ（サイドカー） → SQSから取り出して処理
                                      失敗 → デッドレターキューへ
```

## グレースフルシャットダウン
- コンテナ停止時、処理の途中でいきなり止まるとデータが壊れることがある
- 「止まれ」と言われたら新規リクエストだけ受け付けをやめ、処理中のタスクが終わってから止まる
- デプロイ更新で安全にコンテナを入れ替えるために必須

## 関連概念
- cloud_infrastructure（ECS・SQSとの組み合わせ）
- separation_of_concerns（API処理とWorker処理を分離する思想）
